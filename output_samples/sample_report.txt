  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ██ CRITICAL #1 of 4 ██                                                │
  │  Exposed Patient Data File in Cloud Storage                            │
  │  Finding ID: dork-scan-0001  |  TIPPSS: Privacy, Protection, Security  │
  └─────────────────────────────────────────────────────────────────────────┘

  WHAT WAS FOUND (Plain Language):
    A file containing patient information was found stored in a cloud
    storage location (Amazon S3 bucket) that is publicly accessible —
    meaning anyone on the internet can download it without logging in
    or needing any credentials.

  TECHNICAL DETAIL:
    A Google dork search returned a publicly accessible CSV file at
    s3://example-rmc-ai-data/training/patient_cohort_2024.csv
    The file appears to be an AI model training dataset containing
    patient identifiers, dates of birth, diagnoses, and medication data.

  BUSINESS IMPACT:
    → Potential HIPAA breach notification obligation (45 CFR §164.400)
    → If 500 or more patients affected: HHS notification required within
      60 days and media notification may be required
    → Average cost of a healthcare data breach: $10.9 million
    → Reputational damage if breach becomes public knowledge
    → Possible OCR investigation and civil monetary penalties

  WHO SHOULD ACT:
    Security Team (immediate), Compliance Officer, Legal Counsel,
    Privacy Officer

  REMEDIATION STEPS:
    1. Immediately make the S3 bucket private (AWS Console → Block
       Public Access → Enable for this bucket)
    2. Enable S3 access logging to determine if the file was accessed
       and by whom
    3. Preserve evidence — do not delete the file until legal counsel
       advises (it may be needed for breach investigation)
    4. Contact Legal Counsel and Privacy Officer TODAY to begin HIPAA
       breach risk assessment
    5. Conduct full audit of all S3 buckets for similar exposures
    6. Implement AWS S3 Block Public Access at account level

  RESPONSE DEADLINE:  IMMEDIATE — Today

  ─────────────────────────────────────────────────────────────────────────

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ██ CRITICAL #2 of 4 ██                                                │
  │  AI System Password Key Exposed in Public Code Repository              │
  │  Finding ID: github-intel-0001  |  TIPPSS: Identity, Security          │
  └─────────────────────────────────────────────────────────────────────────┘

  WHAT WAS FOUND (Plain Language):
    A secret password key used to authenticate ERMC's AI system was found
    posted in a publicly visible code repository on GitHub. This is
    similar to leaving the master key to a building taped to the front
    door — anyone who finds it can use it.

  TECHNICAL DETAIL:
    A GitHub code search returned an OpenAI API key (OPENAI_API_KEY)
    committed in a .env configuration file in the repository
    github.com/example-rmc/clinical-nlp. The key was present in the
    repository's commit history and may have been accessible for an
    unknown period of time.

  BUSINESS IMPACT:
    → Any person who found this key can use ERMC's AI services, billed
      to ERMC's account — potentially incurring thousands of dollars
      in unauthorized charges
    → If the AI system was used to process patient data, the exposed
      key may allow unauthorized access to patient query history
    → The key may grant access to custom AI models fine-tuned on
      ERMC clinical data
    → GitHub indexes repositories continuously — the key may already
      have been discovered and used

  WHO SHOULD ACT:
    Security Team (immediate), IT Leadership

  REMEDIATION STEPS:
    1. IMMEDIATELY rotate (replace) the exposed API key in the OpenAI
       console — the old key will stop working and the risk is contained
    2. Check OpenAI usage logs for any unauthorized API calls made
       using the exposed key
    3. Remove the key from the GitHub repository's entire commit history
       (requires a tool called git-filter-repo — IT can perform this)
    4. Enable GitHub secret scanning to prevent future key exposure
    5. Store all API keys in a secure vault system going forward —
       never in code files

  RESPONSE DEADLINE:  IMMEDIATE — Within the hour if possible

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ██ CRITICAL #3 of 4 ██                                                │
  │  Clinical AI System Accessible from Open Internet Without Password     │
  │  Finding ID: shodan-scan-0001  |  TIPPSS: Identity, Protection,        │
  │                                           Safety, Security             │
  └─────────────────────────────────────────────────────────────────────────┘

  WHAT WAS FOUND (Plain Language):
    ERMC's AI language model server — the system that powers clinical AI
    tools — is accessible from the open internet without requiring any
    login or password. Anyone who knows the address can connect to it,
    ask it questions, or attempt to manipulate it.

  TECHNICAL DETAIL:
    Shodan (an internet infrastructure search engine) reports port 11434
    (Ollama LLM server) open and publicly accessible on IP 203.0.113.42
    (hostname: ai.example-rmc.org). The Ollama API requires no
    authentication by default. This port is listed publicly in Shodan's
    index, meaning it is discoverable by any attacker using standard
    reconnaissance tools.

  BUSINESS IMPACT:
    → Unauthorized parties can query ERMC's clinical AI system directly
    → Attackers can attempt prompt injection — crafting inputs designed
      to manipulate the AI into revealing confidential information or
      producing harmful clinical guidance
    → If the AI has access to patient scheduling, records, or clinical
      decision support, manipulation of its outputs could affect patient
      care
    → The AI server may contain custom models trained on ERMC patient
      data, which could be extracted

  WHO SHOULD ACT:
    Security Team, Network/Infrastructure Team (immediate)

  REMEDIATION STEPS:
    1. Configure the Ollama server to listen on localhost (127.0.0.1)
       only — this takes one configuration change and immediately
       closes the public exposure
    2. If remote access to the AI server is required, place it behind
       a VPN or authenticated reverse proxy (do not expose directly)
    3. Review server access logs for any unauthorized connections
    4. Conduct an audit of what data the AI server can access and
       whether that access is appropriately restricted

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  ██ CRITICAL #4 of 4 ██                                                │
  │  AI Development Workspace Open to Internet — Possible PHI Exposure     │
  │  Finding ID: dork-scan-0002  |  TIPPSS: Identity, Privacy,             │
  │                                          Protection, Security          │
  └─────────────────────────────────────────────────────────────────────────┘

  WHAT WAS FOUND (Plain Language):
    A technical workspace where ERMC staff develop and test AI tools
    (a Jupyter notebook server) is accessible from the open internet
    without a password. These workspaces often contain patient data
    samples, API credentials, and the code that powers ERMC's AI
    systems.

  TECHNICAL DETAIL:
    A passive probe and Google dork search found a publicly accessible
    Jupyter notebook server at jupyter.example-rmc.org (port 8888)
    without authentication. The server's open directory listing was
    visible, showing notebooks including clinical_model_v2.ipynb and
    patient_cohort_analysis.ipynb — titles suggesting patient data
    may be present in the workspace.

  BUSINESS IMPACT:
    → Full contents of the development workspace are readable and
      downloadable by anyone on the internet
    → Patient data in notebooks = potential HIPAA breach
    → API keys and credentials found in notebook code can be used
      to access ERMC systems
    → Attackers can execute code on the server, potentially pivoting
      to other ERMC systems

  WHO SHOULD ACT:
    Security Team, AI/Data Science Team (immediate)

  REMEDIATION STEPS:
    1. Immediately take the Jupyter server offline or block all
       external access at the firewall
    2. Enable Jupyter authentication (token or password) before
       bringing it back online
    3. Audit all notebook files for patient data and API credentials
    4. Move all development environments to ERMC's internal network —
       development servers should never be internet-facing
    5. Assess whether any patient data in notebooks constitutes
       a HIPAA breach requiring notification
